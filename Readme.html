Domn

The Domn table contains definitions for various integer IDs used within other tables. For example, entity types, institution types, entity status types, service types, transaction types, and so on. Unlike other tables within the database, the Domn table should be treated as read-only (since it contains values that should be treated as constants).

Holder

Holders are individuals who have ownership over entities within the database (e.g. Holder ‘Mukesh Palaknakh’ owns an insurance policy). This table is modified by the ALRv2 application when holders are added, modified or deleted. Holder_IDs are unique, and are assigned upon holder creation. Only the Holder_ID and Holder_Name fields are mandatory when creating a holder.

Only the Institution_ID, Code and Name fields are required to create an institution.

EntityHistory

This table keeps track of the history of a given entity. For any reason, funds may be transferred between entities. Perhaps savings from a given savings account were used to purchase shares, or a savings account at one bank was closed and the funds were moved to another bank with a better interest rate. Whatever the reason, entity history has been created. In this table, ‘Entity_ID’ represents the child (recipient) entity whilst ‘Linked_Entity_ID’ represents the parent (provider) entity. This table is modified by the ALRv2 application when the user chooses to add, modify or delete entity history from a given entity. Neither column within this table is unique as a given entity can have more than one parent or more than one child. Both fields are required to create an entity history relationship.

EntityShrTxn

This table keeps track of all transactions that occur for a given set of shares or managed funds. Such transactions include the purchasing of shares, the selling of shares, the payment of dividends and dividend reinvestment plans (all of which are described within the ‘Domn’ table under the ‘DomnName’ of ‘ShareTxnType’). This table includes a variety of fields, not all of which are used depending on the transaction type (e.g. capital gains tax is only relevant when shares have been sold). Fields irrelevant to the type of transaction being stored are expected to be populated with zero values. This table is modified by the ALRv2 application when a user chooses to add, modify or delete transactions associated with a particular ‘shares/managed funds’ entity. This table is also used to calculate the current value of the entity. All fields except the transaction description are required to create a valid entry within this table.

Debugging Cheat Sheet

MariaDB provides the backend database functionality for ALRv2. MariaDB installation instructions and database locations on other operating systems will vary. Below are some basic commands that can be used with MariaDB in order to assist any potential future debugging.

Command

Description

mysql.server start

Starts the MariaDB Service

mysqladmin shutdown

Stops the MariaDB Service

CREATE USER 'NAME_OF_USER'@'localhost' IDENTIFIED BY 'ADD_A_PASSWORD';

Create a user and set a password for them

SELECT host, user, password FROM mysql.user;

List all users

GRANT ALL privileges ON `ALR_DB`.* TO 'RaviBarar'@localhost;

Grant all privileges to a given user

CREATE DATABASE DatabaseName;  

Creates a database

SHOW DATABASES;

Lists all databases

USE database_name;

Selects a database for work

CREATE TABLE Book(id INT NOT NULL AUTO_INCREMENT, name VARCHAR(100) NOT NULL, PRIMARY KEY (id));

Creates a table within the selected database. name, then data type followed by optional modifiers. NOT NULL is self explanatory. AUTO_INCREMENT will automatically increment the id whenever a new entry is added. PRIMARY KEY selects the column to be marked as primary.

SHOW TABLES;

Lists all tables within a database

DESC TableName;

Describes a given table within a database

INSERT INTO book (id, name) VALUES(1, 'MariaDB Book’), (2, ‘Example2’);

Inserts multiple records into the table, populating the specified columns with the data specified within that same order.

SELECT * FROM book;

Selects all records from the table book.

UPDATE book SET name=‘Blah’, author=‘haha’ WHERE id = 1;

Updates a selected record with new column information given a condition (in this case, the id number)

Select * FROM book WHERE author = “Author2” AND name = “ass”;

Selects all records within the table book where the author is Author2 and the name of the book is ass. OR can also be used instead of AND.

SHOW CREATE TABLE Tablename;

Output the SQL instructions used to recreate the table

ALTER TABLE Tablename ADD PRIMARY KEY(id)

Add the column ‘id’ as the primary key for the table

ALTER TABLE Tablename MODIFY COLUMN Columnname <datatype> <conditions>

Modify an existing column’s datatype or conditions (e.g. NOT NULL, auto_increment, etc.)

ALTER TABLE TargetTable ADD CONSTRAINT FK_TargetTable_SourceTable_ColumnName FOREIGN KEY (ColumnName) REFERENCES SourceTable(ColumnName);

Creates a foreign key linking the specified column in the target table with the specified column in the source table.

ALTER TABLE Tablename DROP FOREIGN KEY FK_Name;

Deletes the foreign key ‘FK_Name’ from the specified table.

mysqldump -u username -p database_name > data-dump.sql

Requests the user’s password before backing up the database to the selected path as a SQL file. Note this command should be run OUTSIDE the MariaDB shell.

mysql new_database < data-dump.sql

Restores a backup of a database. Note that the new_database should be created before running this import command. Note that a user will have to be granted privileges on this database again. Note this command should be run OUTSIDE the MariaDB shell.
